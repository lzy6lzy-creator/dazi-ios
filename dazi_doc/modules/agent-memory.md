# Agent 与 Memory 设计

最后更新：2026-06-05

## 1. 角色定位

AI 是用户的搭子经纪人，不是通用聊天机器人。它的工作是帮助用户把模糊的活动想法整理成可匹配的 Event，并在匹配后帮助双方更轻松地沟通。

核心原则：

- 不替用户承诺、报名或做最终决定。
- 不夸大用户优势，不隐藏限制条件。
- 不泄露用户私有 Memory 给对方用户或对方 AI。
- 对话要短、自然、可执行，优先推进活动创建和沟通。

## 2. 工作模式

| 模式 | 触发 | 主要输出 |
| --- | --- | --- |
| 用户对话 | 用户在 AI 页发送消息 | 问候、澄清、事件草稿、编辑建议 |
| 事件抽取 | 对话已足够或需要兜底解析 | `EVENT_DRAFT` / 结构化 Event |
| 澄清卡片 | 缺少关键字段且适合结构化追问 | 选项卡、输入建议、上下文回复 |
| A2A 精排 | 匹配候选进入精排 | compatibility、理由、风险 |
| 聊天室协助 | 用户在聊天室 @AI | 基于房间、事件、用户上下文的回复 |

## 3. 事件创建协议

用户表达活动意图后，AI 需要收集：

- 活动类型：吃饭、咖啡、看展、徒步、旅行、电影等。
- 时间：开始、结束或自然语言时间窗口。
- 地点：城市、商圈、地标、区域、线上或不限地点。
- 偏好：预算、人群、节奏、活动风格。
- 限制：不可接受条件、安全边界、特殊要求。

信息足够时，AI 产出事件草稿；用户确认后才创建 Event。服务端负责从回复中提取结构化字段，避免客户端直接信任自然语言。

## 4. 澄清策略

优先级：

1. 时间和地点会影响匹配池，缺失时优先澄清。
2. 活动类型不明确时先澄清类型。
3. 偏好和限制可以边聊边补，不阻塞所有活动创建。
4. 用户已经表达明确取消意图时，不继续推进事件。

澄清卡片适合用于有限选项，例如时间段、预算、地点范围；开放问题适合用于用户偏好描述。

## 5. Memory 类型

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| preference | 稳定偏好 | 喜欢安静咖啡馆、偏好周末下午 |
| constraint | 限制条件 | 不喝酒、不能太晚回家 |
| behavior | 行为习惯 | 经常临时改时间、回复较慢 |
| feedback | 匹配反馈 | 不喜欢太强势的搭子 |

Memory 不是用户公开资料。它只作为 AI 生成回复和匹配判断的辅助信号。

## 6. 分层记忆

当前后端已引入分层记忆：

- `agent_memories`：用户级长期记忆，适合稳定偏好和限制。
- `event_memories`：事件级记忆，适合一次活动里的临时偏好。
- `memory_evidence`：记忆证据，用于解释来源和更新置信度。

写入原则：

- 明确、可复用的信息才写长期记忆。
- 一次性活动细节优先进入事件记忆。
- 低置信度记忆不应强行影响关键匹配。
- 用户否定或纠正后要降权、更新或停用旧记忆。

## 7. 安全边界

- 文档和代码不得保存真实 API key。
- Prompt 不要求模型输出 chain-of-thought。
- 对外回复不展示内部标签、隐藏控制 token 或推理草稿。
- 聊天室中 AI 回复需要知道 @ 来源，但不能越权读取另一方私有 Memory。
