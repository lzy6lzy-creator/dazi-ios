# i搭不搭 PRD

最后更新：2026-06-05

## 1. 产品定位

`i搭不搭` 是以具体活动意图为中心的 AI 搭子匹配产品。用户不浏览人，也不先找陌生人聊天，而是先表达“我想做什么”，再由系统为相容的活动意图撮合搭子。

核心范式是 **Event as Primitive**：

```text
传统社交：人 -> 找人 -> 再找事情
i搭不搭：人 -> 想做的事情 -> 找一起做的人
```

## 2. 目标用户

| 用户类型 | 典型状态 | 核心需求 |
| --- | --- | --- |
| 新城市居民 | 刚到一个城市，社交圈未建立 | 找人吃饭、看展、运动、周边游 |
| 轻社交用户 | 不想刷人设，也不想长期尬聊 | 低压力、短流程、可控社交 |
| 兴趣驱动用户 | 有明确活动兴趣和时间窗口 | 找到活动目标真正相近的人 |

早期重点用户是一线和新一线城市中愿意尝试 AI 辅助社交的年轻用户。

## 3. 产品原则

1. **不做人找人**：产品不提供用户搜索、陌生人列表或滑卡浏览。
2. **事件优先**：匹配围绕活动意图发生，用户画像只作为事件匹配的辅助信号。
3. **无普通私聊**：用户之间只在事件聊天室中沟通。
4. **AI 做经纪人，不替用户决定**：AI 可以澄清、总结、建议、协调，但不能替用户承诺或忽略重要约束。
5. **隐私边界清晰**：每个用户的 Memory 只服务自己的 AI 搭子经纪人，不能被对方用户或对方 AI 直接读取。

## 4. 核心概念

### Event

事件是系统最小业务单位。当前字段包括标题、活动类型、城市、地点、开始/结束时间、偏好、限制、状态、向量、匹配轮次等。

### AI 搭子经纪人

每个用户有一个自己的 AI 搭子经纪人，负责理解活动意图、追问缺失信息、生成事件草稿、协助编辑活动、在聊天室中回应 @ 请求。

### Memory

Memory 是 AI 对用户长期偏好和限制的结构化记忆。它用于改善活动创建和匹配，不是公开资料。

### Chat Room

匹配成功或被动邀请接受后创建事件聊天室。聊天室承载匹配摘要、双方投票、消息沟通和 AI 协助。

## 5. 用户主流程

### 5.1 登录与资料

内测阶段使用手机号 + 内测验证码登录。首次登录自动创建用户和默认 AI 搭子经纪人。用户可编辑昵称、头像、城市、兴趣和简介。

### 5.2 创建活动

用户在 AI 对话中自然表达活动意图。AI 收集活动类型、时间、地点、偏好和限制；信息不足时返回澄清问题或结构化澄清卡片；信息足够后生成事件草稿并等待用户确认。

### 5.3 匹配

事件确认后进入待匹配状态。后端生成 embedding，执行状态/时间/地点/黑名单硬过滤，再通过向量召回和 A2A 精排选择候选。主动匹配多轮失败后，系统可以向其他用户发起被动邀请。

### 5.4 聊天室

匹配成功后用户进入聊天室。聊天室优先展示匹配摘要和“搭/不搭”决策，用户可以发送消息、查看投票状态、@ 自己的 AI 或对方 AI 获得上下文协助。

### 5.5 结束与反馈

当前已有聊天室关闭和投票基础能力。正式反馈、活动后评价、推荐质量回流仍属于后续范围。

## 6. 已实现范围

- 内测登录、JWT、用户资料、AI 资料。
- AI 对话、历史恢复、事件草稿、澄清问题、活动编辑。
- Event 创建、编辑、取消、列表、详情。
- embedding 生成、主动匹配、A2A 精排、被动邀请。
- 聊天室、消息、WebSocket、投票、黑名单。
- 管理后台、测试数据、匹配预览、日志查看。
- iOS 和 Android 基础客户端。
- 地点匹配 normalizer/policy 与本地评测。

## 7. 上线前范围

P0 目标是可控内测，不追求完整正式版：

- iOS TestFlight 构建和真机主链路验证。
- 内测白名单、验证码、登录异常链路验证。
- 主动匹配、被动邀请、聊天室投票的端到端验证。
- 部署健康检查、日志观察、基础回滚路径。
- 敏感配置移出文档和代码，只保留在安全位置。

## 8. 非目标

- 不做公开用户广场。
- 不做长期私信系统。
- 不做 AI 自动替用户约定活动。
- 不把 Memory 暴露给对方用户或对方 AI。
- 第一阶段不接地图 API，不强依赖地理坐标。

## 9. 成功指标

- 登录到首次活动创建完成率。
- 活动草稿确认率。
- 创建事件到匹配/邀请的平均耗时。
- 匹配后双方投“搭”的比例。
- 聊天室有效消息数。
- 内测用户次周留存。
