# Conversation Lab

`conversation-lab` is a standalone test bed for the `i搭不搭` main conversation flow.
It does not import or modify the iOS, Android, or server app code. The goal is to
iterate on the conversation prompt and orchestration contract with deterministic
simulated users before deciding what should be merged into the product code.

## What It Tests

The lab models the main chat decision as one orchestrator prompt:

- `chat`: ordinary conversation or vague intent.
- `clarify`: structured question cards, up to 3 questions.
- `draft`: enough information exists to show an activity draft.
- `cancel`: the user clearly abandons publishing.

Publishing itself is intentionally outside the prompt. The simulated product flow
treats publish as a button event after a `draft` action. Post-publish memory should
therefore remain a separate background prompt/process and is not part of this lab.

## Project Layout

- `prompts/orchestrator_v2.md`: prompt contract for the main conversation agent.
- `scenarios/core.json`: core scenario suite and expected outcomes.
- `src/conversation_lab/engine.py`: turn loop, publish-button simulation, state.
- `src/conversation_lab/rule_backend.py`: offline backend approximating the prompt contract.
- `src/conversation_lab/user_agents.py`: deterministic simulated user agent.
- `src/conversation_lab/evaluator.py`: scenario assertions and quality gates.
- `reports/orchestrator_v2-rule-core.json`: latest generated transcript report.

## Run

From this directory:

```bash
python3 -m unittest discover -s tests -q
PYTHONPATH=src python3 -m conversation_lab.run_lab --prompt orchestrator_v2 --backend rule --scenario-set core --write-report
```

Expected result:

- Unit tests pass.
- Scenario output has `"passed": true`.
- A fresh JSON report is written to `reports/orchestrator_v2-rule-core.json`.

## Tag-based Event Extraction Lab

The current product contract is JSON. To test stream-friendly tag output without
changing the product path, run the standalone extraction lab:

```bash
PYTHONPATH=src python3 -m conversation_lab.tag_event_lab --provider all --write-report
```

It compares Kimi/Moonshot and DeepSeek V4 Flash on the same cases, with
`stream=true` by default. Kimi reads `KIMI_API_KEY`, `MOONSHOT_API_KEY`, or
`LLM_API_KEY`. DeepSeek reads `DEEPSEEK_API_KEY`, `DEEPSEEK_BASE_URL`, and
`DEEPSEEK_MODEL`. Reports are written to `reports/tag_event_lab-*.json` and
`reports/tag_event_lab-latest.json`.

## Current Coverage

The core suite covers:

- Food/hotpot flow with city correction and renewed clarification.
- Sports flow that asks for time, skill level, and cost preference.
- Ordinary chat that must not become a draft.
- Mid-flow cancellation.
- Prompt-injection text treated as business data.
- Bar/social drinking flow with age preference clarification.
- Draft revision after the first draft, with latest user condition overriding old state.

## Findings From This Iteration

The lab caught several flow bugs that are easy to miss in manual chat testing:

- Re-asking was blocked because `重新问` was treated as draft revision before pending clarifications were processed.
- Repeated clarification cards could appear without tracking already asked question ids.
- Casual chat could be forced toward publishing by an over-eager simulated user.
- Cancellation after a draft could be swallowed by auto-confirm behavior.
- Draft objects were mutated after being stored in transcripts, hiding earlier states.
- Area answers were misclassified when generic location matching ran before question-id matching.
- Draft revision appended new location to the old one instead of replacing it.
- Old control tokens and operation phrases could leak into prompt wording or draft preferences.

## Merge Recommendation

When this flow is merged into the real app/server, keep the same separation:

- Main chat uses one orchestration prompt to choose `chat`, `clarify`, `draft`, or `cancel`.
- Confirm publish remains a UI/button event, not an LLM decision.
- Event extraction is the `draft` payload generated by the main prompt.
- Background memory runs only after successful publish and uses a separate prompt.
- The app should persist asked question ids per draft session to avoid duplicate cards.
- User revision should merge by intent: replacement language such as `改成` overrides old values.

Before product merge, connect the lab runner to the real model/backend as a second backend
beside `rule`, and require the core scenarios to pass as a release gate.
